## 2026-07-16｜G4 内容溯源字段与 AI HOT 参考边界

### 决策

事件契约增加 `fact_type`、`event_time_precision`、`event_time_source`。MiniMax 必须在限定枚举内返回事实类型；企业官网发布的政策主张默认以发布企业为主体，不能把文中讨论的政府直接写成事件发布主体。RSS/Atom 的精确时间可以继续用于实时窗口和排序，但必须标记为 `rss` 来源，不能伪装成官方页面确认的精确事件时刻。旧 JSONL 事件缺少这些字段时按 `unknown` 兼容读取，不做破坏性迁移。

本轮不实现连续 alpha / beta 预发布版本流的自动降噪。该问题不影响状态正确性，继续由 G4 删除无明确功能、安全或兼容性变化的连续预发布构建；积累真实重复样本后，再评估按仓库、版本通道和实质变更聚合。

### AI HOT 参考结论

数字生命卡兹克的 [AI HOT Skill](https://github.com/KKKKhazix/khazix-skills/tree/main/aihot) 以 MIT 许可证开源；AI HOT 提供无需 Key 的匿名只读 RSS 与 REST API。MVP 后续将其评估为二级“线索发现源”，不作为最终事实依据，也不直接复制其日报作为本项目日报。

优先借鉴和验证的机制包括：宽问题默认使用精选池、精选与全量查询分离、服务端关键词/分类筛选、滚动时间窗与自然日日报分离、先查 fingerprint 再拉取详情、使用 ETag / Last-Modified、保留来源署名和站内 canonical。官方 Skill 明确说明无需 MCP server；目前只确认 Skill 开源和公开 API 可用，未确认 AI HOT 网站后端与筛选算法源码已经开源，因此不把第三方文章对算法的描述视为可直接复用代码。

### 安全边界

AI HOT 若进入采集层，只允许匿名 `GET` 公共端点，返回内容一律按不可信外部输入处理。它不需要新增 Secret；现有 MiniMax/Kimi Key 仍只通过环境变量和 GitHub Actions Secrets 注入，不写入请求样本、日志、状态文件或公开候选包。

## 2026-07-15｜M3 低延迟抽取与 feed 模型调用门槛

### 决策

MiniMax M3 负责事件标准化时固定使用 `thinking: {type: disabled}`、`max_tokens: 512`、`reasoning_split: true` 和 45 秒请求超时。模型调用前先按候选批次的 `lookback_start` 至 `end` 筛选带发布时间的 feed 项，再按来源最多保留 20 条。未标明发布时间的 feed 项不进入本次实时模型队列。

### 依据

2026-07-15 的本地真实验证显示，未关闭思考时，单条抽取在 30 秒内超时；即使限制为 512 输出 Token，思考内容也会耗尽预算而没有完成 JSON。使用 `thinking.disabled` 后，同一类请求以 304 输入 / 105 输出 Token 返回完整结构化 JSON。三条官方 feed 共返回 1,055 条历史项；按实时窗口筛选后，晨间仅 10 条、午间仅 2 条进入模型处理。该门槛防止历史 feed 重放造成不必要的 Token 消耗和重复候选。

### 边界

此配置只适用于默认的高频结构化抽取，不适用于 Kimi 深度仲裁、后续专题分析或 ChatGPT 的定向调研。来源、窗口和每源上限可依据两周真实运行数据调整，但调整前必须保留回归测试和决策记录。

## 2026-07-15｜成功批次水位线与 G3 调度连续性

### 决策

GitHub Actions 仍以北京时间 06:20 和 12:20 启动；首次批次按计划边界回查，之后每一个成功批次以最近一次完整成功批次的完成时间作为下一批正式数据范围起点，并额外保留六小时采集回查。失败和部分成功批次不推进水位线。

### 依据

GitHub Actions 计划触发可能延迟。若只依赖固定的 07:10 / 13:10 边界，06:20 / 12:20 的预留启动时间可能形成时间缺口。以成功批次水位线确定正式范围，允许采集层重叠回查，能在不重复推送的前提下处理延迟与临时失败；事件库仍是重复过滤的唯一依据。

### 边界

水位线不是“事件已失效”的判断，也不是逐来源的永久状态机。它仅是整个候选批次的连续成功边界；来源级别的精细水位线仅在证据显示有必要时再引入。

## 2026-07-15｜G3 云端验证的默认分支门槛

### 决策

PR 分支使用 `push` / `pull_request` 触发的无密钥 `Tests` 工作流完成云端代码验证。真实 `workflow_dispatch`、Secrets 注入、状态写入和定时任务验证，必须在工作流文件已进入默认分支后执行。

### 依据

GitHub 对手动与计划工作流的注册入口以默认分支为准。当前仓库此前没有已注册的工作流；向功能分支推送工作流后，CI 可以运行，但 API 无法按该文件名派发 `workflow_dispatch`，返回 404。把此限制明确写入运行手册，避免把平台限制误判为代码或密钥故障。

## 2026-07-14｜模型套餐能力观察基线与 Kimi 路由

### 决策

MVP 不设置 Kimi 的精确套餐限额变量。MiniMax 是默认模型；Kimi 仅处理会影响是否推送的高优先级歧义项，采用结构化摘要、少量证据和最多三条相似历史记录的顺序调用方式。每批次最多进入三条 Kimi 候选，并记录两模型实际输入、输出 token 和调用结果。

两周真实运行后，根据用量、429、超时与误判情况决定是否需要增加精细限额。收到套餐、模型或提供方政策变化时，先复核本条再调整实现。

### 截至 2026-07-14 的能力参考

## 2026-07-15｜Kimi For Coding 路由实测修正

### 决策

`KIMI_API_KEY` 采用 Kimi For Coding 会员 Key，不使用 Moonshot 开放平台入口。MVP 的 Kimi 条件复核路由固定为 `https://api.kimi.com/coding/v1` 与 `kimi-for-coding`。因该模型只接受 `temperature: 1`，在版本化配置中显式覆盖共享客户端的默认 `0.1`；同时将 `max_tokens` 限为 512。

### 验证证据

2026-07-15 的最小真实仲裁调用返回合法 `new_event`，用量为 291 输入 / 335 输出 token。此前对 `https://api.moonshot.cn/v1` 的 HTTP 401 保留为两类 Key/服务隔离的证据，不再视为 Kimi 条件复核不可用。

### 影响

Kimi 仍仅处理高优先级且会影响推送判断的歧义项，每批最多三条、顺序执行；本次不改变其用量控制边界或 GitHub Actions 的 Secrets 名称。

## 2026-07-15｜MVP 优先、最小必要验收与实现探索空间

### 决策

本项目采用“先形成可用 MVP、再以真实运行数据迭代”的开发风格。阶段验收只保留会影响状态一致性、信息去重、公开交付、密钥安全或用户实际阅读的关键检查；不因非阻塞性的实现偏好、文档表述或可替代的代码形态反复延长阶段。

设计文档描述目标、边界、数据契约和验收结果，不将示例代码或某一种内部实现写成唯一允许方案。只要不破坏既定契约、测试与安全边界，Codex 可以选择更简洁、更可靠或更易维护的实现，并在提交说明中解释关键取舍。

### 依据

当前系统只有一位用户，公开数据与状态均可追溯，风险主要集中于 API Key、重复推送、时间边界和失败保护。过度细化的前置规格与多轮非阻塞审核，会降低实现速度并压缩后续更优方案的探索空间；真实批次的运行结果更适合驱动后续优化。

### 边界

此决策不放宽以下要求：不提交密钥或私人信息；不让失败/部分成功覆盖 `delivery/current/`；不绕过事件库的去重状态；不把未核验的评论伪装为事实；涉及公开目录、数据模型、调度边界或用户可读交付的破坏性变更，仍需提供相应测试与简短验收证据。

## 2026-07-15｜成功交付语义与手动回补可见性

### 决策

`data/events/` 记录“系统曾观察到”的公开事件，失败与部分成功批次也可写入，以保留诊断证据；但后续重复抑制只读取此前已经进入成功 GitHub 候选包的 `data/gpt-handoffs/`。因此，一个事件不会因只出现过失败或部分成功批次而失去第一次交付机会。

每份 manifest 增加触发和覆盖语义：`trigger_type`、`coverage_mode`、`scheduled_for`、`actual_started_at`、`is_backfill`、`is_zero_length_window`。其中 `scheduled_for` 表示 GitHub Actions 计划启动时刻（晨间 06:20、午间 12:20），计划数据截止仍由 `data_range.end` 表示（晨间 07:10、午间 13:10）。候选 JSON 在根层提供最小 `duplicate_audit`，而 `counts` 保持纯数值以便机器读取。

### 依据

G4 首轮只读审核发现，G3 的迟到手动午间验证按上次成功水位线计算出零长度正式窗口；同时旧实现会从全量观察日志中召回精确匹配事件。前者容易被误读为正常增量，后者可能让未成功交付过的事件直接被抑制。上述修复区分“观察”“成功候选交付”和“回补运行”，但不引入超出 MVP 范围的数据库或运行时写回。

## 2026-07-14｜G2 验收后的事件时间、召回与 GPT 交接修复

### 决策

接受 G2 只读验收提出的四项阻塞性修复：

- 四段媒体回声窗口分别控制召回：热/温期可按事件链召回，冷/归档期只保留精确标识；
- 未知 `event_at` 保持为空并标记 `uncertain`，不得以文章 `published_at` 或发现时间替代；
- 流水线根据采集和标准化错误产生 `success`、`partial` 或 `failed`，仅 `success` 可更新 `delivery/current/`；
- `recent-events.json` 改为最近 30 天已进入成功 GitHub 候选包的跨批次交接历史。

manifest 不再使用含义模糊的 `git_commit_sha`。它记录 `source_commit_sha` 和 `workflow_run_id`；实际输出提交由 GitHub 中该 manifest 文件的提交元数据确定，避免在文件中写入不可能自指准确的提交 SHA。

### 依据

这使 GitHub 保持唯一状态中心：事件时间不会被伪造，冷历史不会不必要地进入模型比较，失败批次不会覆盖可读入口，GPT 也能看到先前已提交审核的事件。
