# Personal Intelligence Briefing：MVP 设计规格

**日期：** 2026-07-14  
**状态：** 已按用户审阅意见更新，待复核后制定实施计划
**仓库：** `Ming-Sir-69/personal-intelligence-briefing`

## 1. 目标与问题

本项目是个人 AI 行业情报与持续学习系统。它不以收集更多链接为目标，而是在有限阅读时间内交付真正新增、可核验、与 AI 产品工作相关的信息。

当前人工或单次 ChatGPT 搜索存在四类问题：

1. 同一事件被不同媒体、语言或标题重复报道；
2. 文章发布时间、事件实际发生时间与系统发现时间混淆；
3. 定时任务缺少可审计的跨次状态，无法可靠判断“是否已报过”；
4. 实时 AI 情报、长期学习和评论性内容混在同一份日报中。

MVP 要建立一个可追溯的事件状态中心，使采集可以为防漏报而重叠，最终推送却不产生无意义重复。

## 2. 使用者与信息需求

使用者是 AI 产品经理，关注 AI 模型、Agent、Agentic Coding、企业软件与制造业数字化。实时信息主要服务于技术边界判断、产品设计、工具选型、发布地区和合规约束判断。

长期学习主题保留在产品路线图中，但不进入 MVP 实时流水线：

- 组织机制、数字化工具与权力关系；
- 宏观经济与国际政治经济；
- 脑科学、学习和自我效率。

这些主题后续按周专题、音频推荐或每日极简自检接入，不用“每日找新内容”的机制填充。

MVP 会先实现它们的**编排规则与报告占位接口**，不自动大规模采集或生成长内容。这样可先验证 AI 实时雷达的去重质量，同时避免后续加入学习模块时破坏报告结构。

## 3. 方案选择

评估过三种方案：

| 方案 | 优点 | 局限 | 决定 |
| --- | --- | --- | --- |
| A. 纯 ChatGPT 计划任务 | 配置快、交付直接 | 状态与去重不可审计，难以长期稳定 | 不采用 |
| B. GitHub 信息底座 + ChatGPT 二次审稿 | 状态可版本化、云端运行、同时保留高质量最终判断 | 需要维护小型流水线 | **采用** |
| C. GitHub 完整自动化交付 | 一致且自动化程度高 | 会失去订阅内 GPT 的最终核验与个性化解释 | 暂不采用 |

方案 B 的职责边界是非对称的：GitHub 层负责覆盖、状态和候选包；ChatGPT 层负责审稿、定向查漏、排序与最终中文交付。ChatGPT 不重新做无限范围的全网检索。

## 4. MVP 范围

### 4.1 必须支持

- 前沿模型发布与实质能力变化；
- Claude Code、Codex、Agentic Coding 工具；
- API 价格、访问权限、上下文窗口、工具调用与地区开放；
- 重要 AI 公司动态；
- 影响 AI 产品的监管、出口控制与地区政策；
- 高价值 GitHub 项目；
- 重要基础研究；
- Apple 端侧 AI 与开发生态。
- 一份版本化的周内学习编排配置：按星期、时段、上次接触时间和内容库存选择一个轻量学习槽位。

### 4.2 MVP 不做

- PDF、Word、HTML 阅读页；
- 向量数据库、复杂网页、用户登录或反馈写回 API；
- 私人工作日志、客户资料、个人行为数据；
- 将长播客、博主评论直接当作实时事实；
- 自动修改 ChatGPT 计划任务；
- 同时对每一条候选调用两个模型。

交付格式仅为 Git 仓库内的结构化 JSON/JSONL 和 Markdown 候选报告。学习槽位只输出题目、链接、时长、认知动作和复习标签，不保存私人生活记录。

## 5. 总体架构

```text
公开来源
  ↓
GitHub Actions（定时或手动运行）
  ├─ 采集、URL 规范化、内容哈希
  ├─ MiniMax 标准化抽取与初步语义判断
  ├─ 历史事件库查询与状态更新
  ├─ Kimi 条件复核疑难事件
  └─ 写入候选包、运行记录和 Markdown 初稿
  ↓
公开 GitHub 固定文件
  ↓
ChatGPT 计划任务（07:30 / 13:30，北京时间）
  ├─ 验证批次新鲜度和状态
  ├─ 审核高影响或不确定候选
  ├─ 定向查漏高优先级官方来源
  └─ 输出最终中文晨报或午报
```

GitHub 仓库是唯一事实来源：它保存事件、来源、水位线、运行历史和候选输出。模型只解决语义理解问题，不承担长期记忆。

## 6. 模型分工

### 6.1 MiniMax（默认模型）

GitHub Actions 从 Secret `MINIMAX_FOR_CODING_API_KEY` 读取密钥。MiniMax 负责：

- 标准化事件字段抽取；
- 主题分类、日期识别、重要性与可信度初评；
- 事件指纹生成；
- 普通语义去重；
- 候选摘要和初步 Markdown。

### 6.2 Kimi（条件复核）

GitHub Actions 从 Secret `KIMI_API_KEY` 读取密钥。仅在任一条件满足时调用 Kimi：

- MiniMax 置信度低于 `0.80`；
- 重大模型、工具、价格、地区或监管事件；
- 日期或来源冲突；
- 相似事件处于模糊区间；
- 疑似“后续实质更新”。

Kimi 失败时事件进入 `uncertain`，不得强行生成结论。

## 7. 时间与批次

用户工作时间为北京时间 08:00—18:00，最终交付目标为：

- 晨报：07:30；
- 午报：13:30。

GitHub Actions 应预留调度延迟：晨间任务约 06:20 启动，午间任务约 12:20 启动。工作流不依赖 Mac mini 在线。

每个事件至少保存：

| 字段 | 含义 |
| --- | --- |
| `event_at` | 事件实际发生或实质更新的时间，优先官方来源 |
| `published_at` | 当前来源发布时间 |
| `discovered_at` | 当前批次发现时间 |
| `first_seen_at` | 系统首次发现时间 |
| `last_seen_at` | 最近再次发现时间 |
| `submitted_to_gpt_at` | 进入 GPT 审核候选包的时间 |
| `reported_at` | 首次最终推送时间，MVP 可为空 |
| `updated_at` | 最近实质更新的时间 |

原则是：**采集窗口允许重叠，报告不允许无意义重复。** 重叠回查用于抵抗 RSS、搜索索引和发现延迟；是否推送由事件状态决定，而不是只由时间窗口决定。

## 8. 分层存储、命名与归档

仓库必须同时提供“固定入口”和“不可变历史”，不能用一个同名文件覆盖所有历史结果。

- `delivery/current/` 是固定入口：供 ChatGPT 计划任务读取，允许每次成功批次覆盖；
- `delivery/archive/YYYY/MM/DD/<batch-id>/` 是不可变批次快照：供回溯、比较和人工验收；
- `data/events/YYYY/MM/` 是按月分区的事件状态；
- `data/runs/YYYY/MM/` 保存每次运行的元数据；
- `data/indexes/` 是可重新生成的活动索引，不是真实历史的唯一副本。

所有时间使用 `Asia/Shanghai`，字段采用带时区的 ISO 8601，文件名采用可排序、不含冒号的形式：

```text
日期目录：2026/07/14
批次 ID：morning-20260714T062000+0800
午间批次：noon-20260714T122000+0800
月度事件文件：events-2026-07.jsonl
单次运行文件：run-morning-20260714T062000+0800.json
```

`current` 中的固定文件名仍为 `manifest.json`、`morning-candidates.json`、`noon-candidates.json` 等；它们必须在 `manifest.json` 中指向对应的不可变 `archive` 路径和 Git commit SHA。ChatGPT 因此可稳定读取最新入口，而人可以回到历史快照验证“某天当时到底看到了什么”。

完整目录、每类文件的写入策略和保留边界见 `docs/architecture/repository-layout.md`。

## 9. 去重时间窗口与历史淡出

事件不会无限期进入同等强度的去重比较。MVP 把窗口配置化，并使用以下**可调整的起始值**：

| 层级 | 默认窗口 | 行为 |
| --- | ---: | --- |
| 热窗口 | 14 天 | 严格比较；同一事件默认视为重复，除非有可核验的实质更新 |
| 温窗口 | 15—90 天 | 只召回同主体/对象的相近事件，辅助识别“后续实质更新”或相关但不同事件，不直接压制新事件 |
| 冷窗口 | 90 天后 | 保留在月度归档中供人工追溯，但不参与日常自动语义去重 |

这不是对所有来源的永久断言。模型发布、API/地区政策、GitHub Trending、论文和监管消息的媒体回声周期不同，因此规则必须写在版本化配置中，记录生效日期、来源类别与修改原因。连续运行两周后，应根据真实重复率、迟到发现率和用户“无效/遗漏”反馈调整窗口；用户也将用外部调研结果校正首轮配置。

## 10. 周内学习编排与认知负荷规则

实时 AI 雷达每天两次运行；长期学习不按同样逻辑强塞“新内容”。学习编排的目标是让输入、提取、迁移和休息分离，而不是在同一天反复讲同一主题。

### 10.1 时段边界

| 时段 | 内容 | 规则 |
| --- | --- | --- |
| 07:30 晨报 | AI 实时雷达 + 一个轻量学习槽位 | 5—10 分钟；提供方向、问题或短材料，不承担深度阅读 |
| 13:30 午报 | 仅 AI 实时增量 | 最多约 3 分钟；不插入长期学习，避免重复和认知切换 |
| 通勤/跑步 | 宏观或产业音频 | 每周 2 次；只给单集、时长与收听目的，不要求即时做长笔记 |
| 晚间较早时段 | 主动回忆或小输出 | 5—20 分钟；优先复习另一日已接触的内容，而非重播当天同一主题 |
| 深度学习后 | 低刺激休息 | 建议 5—10 分钟不刷短视频或继续高密度输入；这是可选行为提醒，不是医疗建议 |

### 10.2 默认周循环

| 星期 | 晨间轻量主题 | 零碎时间输入 | 晚间认知动作 |
| --- | --- | --- |
| 周一 | 组织机制与数字化案例 | 无强制 | 回忆上周宏观/产业内容，并写一条产品映射 |
| 周二 | 宏观框架的一个问题 | 通勤或跑步听宏观金融音频 | 为周一案例补一张因果关系图 |
| 周三 | 脑科学自我校准 | 无强制 | 只做短回忆或休息，不新增重内容 |
| 周四 | AI 进入组织、岗位或流程案例 | 无强制 | 提取周二音频的“事实—解释—不确定性”三句话 |
| 周五 | 产业、政策或地区策略问题 | 通勤或跑步听产业/地缘政治音频 | 回忆周四案例，形成一条设计原则 |
| 周六 | 兴趣驱动的深度主题 | 可选长音频/视频 | 自由输出，不硬性安排 |
| 周日 | 一周关联总结 | 下周待听清单 | 间隔复习与下周选择，不新增实时学习任务 |

同一材料在一个自然日内最多出现一次“输入”槽位；再次出现只能标记为 `spaced_review`，且必须采用不同动作（回忆、因果图、工作迁移或事实/观点拆分），不能伪装成新内容。

该配置以主动提取、间隔复习、学习后减少干扰、睡眠相关巩固为一般学习原则；它不声称人人共享固定的“黄金时段”，个人精力规律只在未来私有反馈层经过观察后调整。

### 10.3 证据边界

本设计只把研究较稳定地支持的方向转为轻量行为约束：主动提取相较单纯重读可促进概念学习；学习后安静休息可能有助于记忆巩固；睡眠与记忆巩固相关。它们不构成医学建议，也不支持为个人规定统一作息。实现和报告中应使用“可尝试”“可观察”而非保证效果的措辞。

- [Karpicke & Blunt（2011）：主动提取与概念学习](https://pubmed.ncbi.nlm.nih.gov/21252317/)
- [Wamsley（2019）：清醒休息与记忆巩固综述](https://pubmed.ncbi.nlm.nih.gov/30683602/)
- [Newman et al.（2019）：睡眠与记忆巩固的元分析](https://link.springer.com/article/10.3758/s13423-018-1528-4)

## 11. 事件模型与状态

事件指纹的逻辑组成是：

`主体 + 对象或产品 + 动作类型 + 官方事件日期 + 核心变化`

例如：`OpenAI + Codex + 发布更新 + 2026-07-14 + 新增并行 Agent 能力`。

每个候选最终只能处于以下一种状态：

| 状态 | 含义 | 是否提交给 GPT |
| --- | --- | --- |
| `new_event` | 新事件 | 是 |
| `substantive_update` | 已知事件的实质更新 | 是，必须标记 |
| `duplicate` | 同一事件的重复报道 | 否 |
| `late_discovery` | 迟到发现的重要旧事件 | 仅重大事件 |
| `related_but_distinct` | 相关但独立的事件 | 按重要性判断 |
| `background` | 长期背景知识 | 否，进入后续学习池 |
| `commentary` | 评论或解读 | 否，进入后续学习池 |
| `uncertain` | 证据或时间不足 | 否 |
| `retracted` | 官方撤回或修正 | 是 |

同一事件再次推送必须有可验证的实质变化，例如预告转正式发布、API 开放、价格/地区/权限变化、关键功能新增、严重事故、安全漏洞或官方撤回。换标题、换媒体、换语言和重复解读不构成实质更新。

## 12. 去重与核验

去重分四层：

1. **URL 级：** 移除 UTM、锚点、移动端差异和重定向差异；
2. **内容级：** 用正文哈希处理完全或高度相同的转载；
3. **事件指纹：** 用标准主体、对象、动作、日期与变化归并；
4. **语义级：** 仅将新候选与历史中召回的少数近似事件交给模型判断。

事实来源分级：

- 一级：官方博客、文档、Release Notes、官方 GitHub、原始论文、政府与监管文件、企业公告；
- 二级：AI 简报、GitHub Trending、Hugging Face Daily Papers 等候选发现渠道；
- 三级：博主、视频、播客和评论文章，仅供长期学习或解释。

二级来源只能发现线索，实时事实必须尽量回到一级来源；不能核验时间或核心事实的候选默认不推送。

## 13. 固定交付接口

ChatGPT 不扫描整个仓库，只读取以下稳定入口：

```text
delivery/current/manifest.json
delivery/current/morning-candidates.json
delivery/current/noon-candidates.json
delivery/current/recent-events.json
delivery/current/morning-preliminary.md
delivery/current/noon-preliminary.md
```

`manifest.json` 必须包含批次 ID、报告类型、开始/完成时间、状态、数据范围、候选数、去重数、疑难数、输出路径、提交 SHA、上游错误和部分成功标记。

如果最新批次失败、过期或不完整，ChatGPT 必须说明上游状态，不能将旧报告伪装成今日新增。

## 14. Markdown 候选报告

候选初稿至少含：

1. 今日结论；
2. 必须关注（最多三条）；
3. 值得了解；
4. Agentic Coding 与工具链；
5. 产品、产业与政策影响；
6. 质量说明（候选、合并、历史重复、证据不足、疑难数量）；
7. 来源与核验状态。

最终 ChatGPT 报告可重排或剔除候选，但不能将评论当成事实，也不能为凑篇幅引入旧闻。

## 15. 失败处理与安全约束

- API Key 只存在 GitHub Actions Secrets 或本地 Keychain；不进入源码、配置、报告、测试夹具或日志；
- 采集、模型或写入失败时保留可诊断的运行记录，但不泄露密钥；
- 没有高价值新增时允许生成明确的“无高价值新增”状态；
- 无法判断的事件进入 `uncertain`，宁可少报，不编造；
- 工作流应有并发锁，避免同类批次并行写入状态；
- 自动提交仅在状态或交付物实际变化时执行；
- 公开仓库不保存原始全文、私人反馈、企业或客户资料。

## 16. 质量指标与验收

MVP 的质量不以抓取数量衡量。优先观察：

- 同一事件重复提交率；
- 重大 AI 事件的人工抽查漏报；
- 无效信息比例；
- 一级来源覆盖率；
- 晨报阅读时间 5—10 分钟、午报不超过 3 分钟；
- 错误批次是否可追溯；
- 连续两次运行是否避免重复候选。

首轮验证场景必须覆盖：

1. 同一事件不同标题/媒体归并；
2. 旧闻今日重发排除；
3. 预告转正式发布识别为实质更新；
4. 时间不明确事件不进入实时区；
5. 上游失败不伪装成新报告；
6. MiniMax 或 Kimi 失败时安全降级；
7. 没有有效信息时输出明确状态。

## 17. 第二阶段方向与后续路线图

MVP 稳定运行一至两周后，才评估：组织与数字化专题的自动选题、宏观音频推荐、脑科学自检的私人反馈层、人工反馈闭环、HTML 阅读页、周报、私有上下文仓库及 PDF/Word 导出。

还要记录一个明确的第二阶段方向：ChatGPT 最终审稿可能发现上游规则错误，但普通 ChatGPT GitHub 连接不能作为可靠写入通道。届时可让 Codex 参与结构化审核反馈的处理与回归修复；自动写回若要实现，必须使用受控的单用途反馈接口，且只能写入类似 `feedback/chatgpt-review/` 的隔离目录，不能让 ChatGPT 直接修改事件库、工作流、代码或 Secrets。MVP 不实现这个闭环。
