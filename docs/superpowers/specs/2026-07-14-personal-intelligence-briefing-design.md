# Personal Intelligence Briefing：MVP 设计规格

**日期：** 2026-07-14  
**状态：** 已按用户审阅意见更新，待复核后制定实施计划
**仓库：** `Ming-Sir-69/personal-intelligence-briefing`

## 1. 目标与问题

本项目是个人 AI 行业情报与持续学习系统。它不以收集更多链接为目标，而是在有限阅读时间内交付真正新增、可核验、与 AI 产品工作相关的信息。

当前人工或单次 ChatGPT 搜索存在四类问题：

1. 同一事件被不同媒体、语言或标题重复报道；
2. 文章发布时间、事件实际发生时间与系统发现时间混淆；
3. 定时任务缺少可审计的跨次状态，无法可靠判断“是否已报过”；
4. 实时 AI 情报、长期学习和评论性内容混在同一份日报中。

MVP 要建立一个可追溯的事件状态中心，使采集可以为防漏报而重叠，最终推送却不产生无意义重复。

## 2. 使用者与信息需求

使用者是 AI 产品经理，关注 AI 模型、Agent、Agentic Coding、企业软件与制造业数字化。实时信息主要服务于技术边界判断、产品设计、工具选型、发布地区和合规约束判断。

长期学习主题保留在产品路线图中，但不进入 MVP 实时流水线：

- 组织机制、数字化工具与权力关系；
- 宏观经济与国际政治经济；
- 脑科学、学习和自我效率。

这些主题后续按周专题、音频推荐或每日极简自检接入，不用“每日找新内容”的机制填充。

MVP 会先实现它们的**编排规则与独立任务接口**，不把长期学习内容混入 AI 实时简报。这样可先验证 AI 实时雷达的去重质量，同时避免后续加入学习模块时破坏报告结构。

## 3. 方案选择

评估过三种方案：

| 方案 | 优点 | 局限 | 决定 |
| --- | --- | --- | --- |
| A. 纯 ChatGPT 计划任务 | 配置快、交付直接 | 状态与去重不可审计，难以长期稳定 | 不采用 |
| B. GitHub 信息底座 + ChatGPT 二次审稿 | 状态可版本化、云端运行、同时保留高质量最终判断 | 需要维护小型流水线 | **采用** |
| C. GitHub 完整自动化交付 | 一致且自动化程度高 | 会失去订阅内 GPT 的最终核验与个性化解释 | 暂不采用 |

方案 B 的职责边界是非对称的：GitHub 层负责覆盖、状态和候选包；ChatGPT 层负责审稿、定向查漏、排序与最终中文交付。ChatGPT 不重新做无限范围的全网检索。

## 4. MVP 范围

### 4.1 必须支持

- 前沿模型发布与实质能力变化；
- Claude Code、Codex、Agentic Coding 工具；
- API 价格、访问权限、上下文窗口、工具调用与地区开放；
- 重要 AI 公司动态；
- 影响 AI 产品的监管、出口控制与地区政策；
- 高价值 GitHub 项目；
- 重要基础研究；
- Apple 端侧 AI 与开发生态。
- 一份版本化的周内学习编排配置：按星期、时段、上次接触时间和内容库存选择独立学习任务。

### 4.2 MVP 不做

- PDF、Word、HTML 阅读页；
- 向量数据库、复杂网页、用户登录或反馈写回 API；
- 私人工作日志、客户资料、个人行为数据；
- 将长播客、博主评论直接当作实时事实；
- 自动修改 ChatGPT 计划任务；
- 同时对每一条候选调用两个模型。

交付格式仅为 Git 仓库内的结构化 JSON/JSONL 和 Markdown 候选报告。学习任务只输出题目、直达链接、时长、认知动作和复习标签，不保存私人生活记录。

## 5. 总体架构

```text
公开来源
  ↓
GitHub Actions（定时或手动运行）
  ├─ 采集、URL 规范化、内容哈希
  ├─ MiniMax 标准化抽取与初步语义判断
  ├─ 历史事件库查询与状态更新
  ├─ Kimi 条件复核疑难事件
  └─ 写入候选包、运行记录和 Markdown 初稿
  ↓
公开 GitHub 固定文件
  ↓
ChatGPT 计划任务（07:30 / 13:30，北京时间）
  ├─ 验证批次新鲜度和状态
  ├─ 审核高影响或不确定候选
  ├─ 定向查漏高优先级官方来源
  └─ 输出最终中文晨报或午报
```

GitHub 仓库是唯一事实来源：它保存事件、来源、水位线、运行历史和候选输出。模型只解决语义理解问题，不承担长期记忆。

### 5.1 默认分支约定

保留仓库当前的默认分支名称 `master`，不额外创建名为 `main` 的平行长期分支。任何工作流、脚本和文档都必须以 GitHub 仓库的“当前默认分支”为准：运行时通过 GitHub 元数据或 `origin/HEAD` 发现，不得把 `main` 或 `master` 硬编码为前提。这样即使新的 Agent 没有对话上下文，也能按仓库配置找到正确分支。

默认分支长期保存已合并的代码、版本化规则和 GitHub Actions 产生的状态。`feature/<主题>` 分支只用于一项待审变更；PR 合并后删除该功能分支。MVP 不建立长期 `data`、`temp` 或第二默认分支。

## 6. 模型分工

### 6.1 MiniMax（默认模型）

GitHub Actions 从 Secret `MINIMAX_FOR_CODING_API_KEY` 读取密钥。MiniMax 负责：

- 标准化事件字段抽取；
- 主题分类、日期识别、重要性与可信度初评；
- 事件指纹生成；
- 普通语义去重；
- 候选摘要和初步 Markdown。

### 6.2 Kimi（条件复核与用量治理）

GitHub Actions 从 Secret `KIMI_API_KEY` 读取密钥。Kimi 的 256K 上下文上限只表示单次请求容量，不代表 Token Plan 的月度额度或任意时间段的速率额度；MVP 必须同时约束这三种风险。

Kimi 只处理 MiniMax 的结构化事件、截断后的官方证据片段和最多三条近似历史事件，不接收候选全文。每次调用的硬上限为：输入不超过 6,000 tokens、输出不超过 1,000 tokens、单批次最多两次调用，且同一时刻只允许一个 Kimi 请求执行。两次日报均触发上限时，理论最大消耗约为 `7,000 × 2 × 2 × 30 = 840,000 tokens/月`；实际用量必须以 API 返回的 usage 字段记录到对应的运行记录中。

调用还必须受 GitHub Actions Variables 中的四项用户配置约束：`KIMI_BRIEFING_MONTHLY_TOKEN_LIMIT`、`KIMI_RATE_WINDOW_SECONDS`、`KIMI_MAX_REQUESTS_PER_WINDOW`、`KIMI_MAX_TOKENS_PER_WINDOW`。这些值以 Kimi Token Plan 控制台显示的真实月度与速率限额为准，不在代码中猜测。未配置时，Kimi 路由默认关闭，候选只由 MiniMax 处理；达到月度预算、时间窗请求/Token 预算或收到 `429` 后，本批次停止 Kimi 调用并把必要项目标为 `uncertain`，不进行并发或无上限重试。

只有在预算仍可用，且满足以下任一复核条件时，才进入 Kimi 队列：

- MiniMax 置信度低于 `0.80`；
- 高重要性事件同时存在日期、来源或实质更新判断冲突；
- 相似事件处于模糊区间。

“重大”本身不自动触发 Kimi；它必须伴随不确定性，避免把 Kimi 变成每条重要新闻的第二次常规处理。Kimi 失败时事件进入 `uncertain`，不得强行生成结论。

## 7. 时间与批次

用户工作时间为北京时间 08:00—18:00，最终交付目标为：

- 晨报：07:30；
- 午报：13:30。

GitHub Actions 应预留调度延迟：晨间任务约 06:20 启动，午间任务约 12:20 启动。工作流不依赖 Mac mini 在线。

每个事件至少保存：

| 字段 | 含义 |
| --- | --- |
| `event_at` | 事件实际发生或实质更新的时间，优先官方来源 |
| `published_at` | 当前来源发布时间 |
| `discovered_at` | 当前批次发现时间 |
| `first_seen_at` | 系统首次发现时间 |
| `last_seen_at` | 最近再次发现时间 |
| `submitted_to_gpt_at` | 进入 GPT 审核候选包的时间 |
| `reported_at` | 首次最终推送时间，MVP 可为空 |
| `updated_at` | 最近实质更新的时间 |

原则是：**采集窗口允许重叠，报告不允许无意义重复。** 重叠回查用于抵抗 RSS、搜索索引和发现延迟；是否推送由事件状态决定，而不是只由时间窗口决定。

## 8. 分层存储、命名与归档

仓库必须同时提供“固定入口”和“不可变历史”，不能用一个同名文件覆盖所有历史结果。

- `delivery/current/` 是固定入口：供 ChatGPT 计划任务读取，允许每次成功批次覆盖；
- `delivery/archive/YYYY-MM/<batch-id>/` 是不可变批次快照：供回溯、比较和人工验收；
- `data/events/` 保存按月命名的事件状态文件；
- `data/runs/` 保存每次运行的元数据；
- `data/active-events.json` 是可重新生成的活动索引，不是真实历史的唯一副本。

所有时间使用 `Asia/Shanghai`，字段采用带时区的 ISO 8601，文件名采用可排序、不含冒号的形式：

```text
归档月份目录：2026-07
批次 ID：morning-20260714T062000+0800
午间批次：noon-20260714T122000+0800
月度事件文件：events-2026-07.jsonl
单次运行文件：run-morning-20260714T062000+0800.json
```

`current` 中的固定文件名仍为 `manifest.json`、`morning-candidates.json`、`noon-candidates.json` 等；它们必须在 `manifest.json` 中指向对应的不可变 `archive` 路径和 Git commit SHA。归档只保留“月份 / 批次”两层，批次名本身携带完整日期与时间；ChatGPT 因此可稳定读取最新入口，而人可以回到历史快照验证“某天当时到底看到了什么”。

完整目录、每类文件的写入策略和保留边界见 `docs/architecture/repository-layout.md`。

## 9. 去重时间窗口与历史淡出

窗口描述的是**同一事件被媒体、转载和解读重复传播的回声期**，不是该事件对产品或行业的有效期。事件生命周期可以更长，但长生命周期不应让所有旧事件持续进入模型语义比较。

MVP 把窗口配置化，并使用以下**统一、低负担的起始值**：

| 层级 | 默认窗口 | 行为 |
| --- | ---: | --- |
| 热窗口 | 0—14 天 | URL、正文哈希、实体、事件指纹和模型语义严格比较；无实质变化即为重复 |
| 温窗口 | 15—30 天 | 不做宽泛历史语义搜索；只按同主体、同产品/模型族、同官方 URL、同版本、同仓库、同论文/法规编号或同事件链召回，再判断是否为实质更新 |
| 冷窗口 | 31—60 天 | 不调用模型做常规比较；只查精确标识符、规范化 URL、主体—对象关系和明确例外规则 |
| 归档缓冲期 | 61—90 天 | 不进入模型召回；仅保留永久精确索引和重大例外规则 |
| 历史期 | 超过 90 天 | 不参与日常去重或候选召回；保留历史记录，只有明确 ID 或重大例外才查询 |

以下标识符永久保存，并可在任何窗口通过精确查询命中：官方公告 URL、模型 ID/版本名、产品与功能名、完整 GitHub 仓库名、arXiv ID/DOI、法规或行政文件编号、Apple 系统/SDK/Xcode 版本和官方事件链 ID。

这套统一规则以高频 AI 工具的短媒体回声期为基准，最高只回查 90 天，降低第一版模型负担。价格、地区、权限、监管等长期事件不靠扩大语义窗口处理，而靠结构化字段与“实质更新”例外规则重报。连续运行两周后，应根据真实重复率、迟到发现率和用户“无效/遗漏”反馈调整窗口；分类窗口是后续优化，不是首版前提。

## 10. 周内学习编排与认知负荷规则

实时 AI 雷达每天两次运行；长期学习以独立 ChatGPT 任务交付，不按同样逻辑强塞“新内容”。学习编排的目标是让输入、提取、迁移和休息分离，而不是在同一天反复讲同一主题。

### 10.1 时段边界

| 时段 | 内容 | 规则 |
| --- | --- | --- |
| 07:30 晨报 | AI 实时雷达 | 5—10 分钟；只处理最新 AI 事件与工作影响，不插入长期学习 |
| 13:30 午报 | 仅 AI 实时增量 | 最多约 3 分钟；不插入长期学习，避免重复和认知切换 |
| 周二、周五下班后 | 宏观或产业音频独立任务 | 默认 18:10；只给单集、直达链接、时长与收听目的，不要求即时做长笔记 |
| 周三晚间 | 脑科学自我校准独立任务 | 默认 21:15；一个观察问题和一个可选调整，不新增重内容 |
| 周日晚间 | 一周关联与间隔复习独立任务 | 默认 20:30；复习先前内容并选择下周主题，不重发当周资讯 |
| 其他深度学习后 | 低刺激休息 | 建议 5—10 分钟不刷短视频或继续高密度输入；这是可选行为提醒，不是医疗建议 |

### 10.2 默认周循环

| 独立任务 | 默认时间 | 内容 | 当次动作 |
| --- | --- | --- | --- |
| AI 晨报 | 每日 07:30 | AI 实时情报 | 阅读、判断今日工作影响 |
| AI 午间增量 | 每日 13:30 | 晨报之后的 AI 新增 | 只核对增量，不做长期学习 |
| 宏观音频卡 | 周二、周五 18:10 | 宏观金融；产业/地缘政治 | 一键直达收听，记录事实与解释的区别 |
| 脑科学自检卡 | 周三 21:15 | 睡眠、专注、休息或信息输入 | 观察一个变量，不同时优化多个变量 |
| 组织与数字化案例 | 周一或周四的可用空档 | 组织、绩效、工具与权力 | 形成一条 AI 产品设计原则 |
| 周回顾 | 周日 20:30 | 本周跨主题材料 | 主动回忆、关联和下周选择 |

同一材料在一个自然日内最多出现一次“输入”任务；再次出现只能标记为 `spaced_review`，且必须采用不同动作（回忆、因果图、工作迁移或事实/观点拆分），不能伪装成新内容。音频卡必须提供原始平台的单集直达链接；没有稳定链接时宁可不推荐，不以搜索页替代。

AI 晨报和午报是 MVP 的首个可运行链路。独立学习任务先定义接口与时段，不立即修改现有 ChatGPT 计划任务；只有 `delivery/current/learning/` 能稳定提供相应卡片后，才分别启用音频、自检和周回顾任务。学习卡可以由后续 GitHub Actions 选择器或人工维护来源生成，但最终推送仍由各自独立的 ChatGPT 任务完成。

该配置以主动提取、间隔复习、学习后减少干扰、睡眠相关巩固为一般学习原则；它不声称人人共享固定的“黄金时段”，个人精力规律只在未来私有反馈层经过观察后调整。

### 10.3 证据边界

本设计只把研究较稳定地支持的方向转为轻量行为约束：主动提取相较单纯重读可促进概念学习；学习后安静休息可能有助于记忆巩固；睡眠与记忆巩固相关。它们不构成医学建议，也不支持为个人规定统一作息。实现和报告中应使用“可尝试”“可观察”而非保证效果的措辞。

- [Karpicke & Blunt（2011）：主动提取与概念学习](https://pubmed.ncbi.nlm.nih.gov/21252317/)
- [Wamsley（2019）：清醒休息与记忆巩固综述](https://pubmed.ncbi.nlm.nih.gov/30683602/)
- [Newman et al.（2019）：睡眠与记忆巩固的元分析](https://link.springer.com/article/10.3758/s13423-018-1528-4)

## 11. 事件模型与状态

事件指纹的逻辑组成是：

`主体 + 对象或产品 + 动作类型 + 官方事件日期 + 核心变化`

例如：`OpenAI + Codex + 发布更新 + 2026-07-14 + 新增并行 Agent 能力`。

每个候选最终只能处于以下一种状态：

| 状态 | 含义 | 是否提交给 GPT |
| --- | --- | --- |
| `new_event` | 新事件 | 是 |
| `substantive_update` | 已知事件的实质更新 | 是，必须标记 |
| `duplicate` | 同一事件的重复报道 | 否 |
| `late_discovery` | 迟到发现的重要旧事件 | 仅重大事件 |
| `related_but_distinct` | 相关但独立的事件 | 按重要性判断 |
| `background` | 长期背景知识 | 否，进入后续学习池 |
| `commentary` | 评论或解读 | 否，进入后续学习池 |
| `uncertain` | 证据或时间不足 | 否 |
| `retracted` | 官方撤回或修正 | 是 |

同一事件再次推送必须有可验证的实质变化，例如预告转正式发布、API 开放、价格/地区/权限变化、关键功能新增、严重事故、安全漏洞或官方撤回。换标题、换媒体、换语言和重复解读不构成实质更新。

## 12. 去重与核验

去重分四层：

1. **URL 级：** 移除 UTM、锚点、移动端差异和重定向差异；
2. **内容级：** 用正文哈希处理完全或高度相同的转载；
3. **事件指纹：** 用标准主体、对象、动作、日期与变化归并；
4. **语义级：** 仅将新候选与历史中召回的少数近似事件交给模型判断。

事实来源分级：

- 一级：官方博客、文档、Release Notes、官方 GitHub、原始论文、政府与监管文件、企业公告；
- 二级：AI 简报、GitHub Trending、Hugging Face Daily Papers 等候选发现渠道；
- 三级：博主、视频、播客和评论文章，仅供长期学习或解释。

二级来源只能发现线索，实时事实必须尽量回到一级来源；不能核验时间或核心事实的候选默认不推送。

## 13. 固定交付接口

ChatGPT 不扫描整个仓库，只读取以下稳定入口：

```text
delivery/current/manifest.json
delivery/current/morning-candidates.json
delivery/current/noon-candidates.json
delivery/current/recent-events.json
delivery/current/morning-preliminary.md
delivery/current/noon-preliminary.md
```

`manifest.json` 必须包含批次 ID、报告类型、开始/完成时间、状态、数据范围、候选数、去重数、疑难数、输出路径、提交 SHA、上游错误和部分成功标记。

如果最新批次失败、过期或不完整，ChatGPT 必须说明上游状态，不能将旧报告伪装成今日新增。

## 14. GitHub 候选初稿与 GPT 最终报告的边界

以下七类**候选初稿内容由 GitHub Actions 输出**：程序完成确定性处理，MiniMax 仅做结构化抽取、初步分类和摘要；这份文件是交给 GPT 审核的输入，而不是最终用户简报。

1. 批次状态、时间范围和数据新鲜度；
2. 高优先级待审候选；
3. 其他有效新增候选；
4. Agentic Coding 与工具链候选；
5. 产品、产业与政策候选；
6. `uncertain`、迟到发现和待核验项；
7. 质量说明：抓取、合并、历史重复、证据不足和疑难数量。

ChatGPT 计划任务读取候选包、`recent-events.json` 和 `manifest.json` 后，才在 ChatGPT 对话中输出最终中文报告。最终报告可以生成“今日结论”、删除低价值项、重排优先级、定向查漏高优先级官方源并说明对用户工作的影响；但它不改写 GitHub 事件状态，也不得把评论当作事实或为凑篇幅引入旧闻。

因此，`delivery/current/*-preliminary.md` 的作者是 GitHub Actions；ChatGPT 的最终 Markdown 是计划任务的会话交付物。MVP 不把最终报告回写仓库。

## 15. 失败处理与安全约束

- API Key 只存在 GitHub Actions Secrets 或本地 Keychain；不进入源码、配置、报告、测试夹具或日志；
- 采集、模型或写入失败时保留可诊断的运行记录，但不泄露密钥；
- 没有高价值新增时允许生成明确的“无高价值新增”状态；
- 无法判断的事件进入 `uncertain`，宁可少报，不编造；
- 工作流应有并发锁，避免同类批次并行写入状态；
- 自动提交仅在状态或交付物实际变化时执行；
- 公开仓库不保存原始全文、私人反馈、企业或客户资料。

## 16. 质量指标与验收

MVP 的质量不以抓取数量衡量。优先观察：

- 同一事件重复提交率；
- 重大 AI 事件的人工抽查漏报；
- 无效信息比例；
- 一级来源覆盖率；
- 晨报阅读时间 5—10 分钟、午报不超过 3 分钟；
- 错误批次是否可追溯；
- 连续两次运行是否避免重复候选。

首轮验证场景必须覆盖：

1. 同一事件不同标题/媒体归并；
2. 旧闻今日重发排除；
3. 预告转正式发布识别为实质更新；
4. 时间不明确事件不进入实时区；
5. 上游失败不伪装成新报告；
6. MiniMax 或 Kimi 失败、限额耗尽或收到速率限制时安全降级；
7. Kimi 的单次、批次、时间窗和月度 Token 限额均可验证；
8. 没有有效信息时输出明确状态。

## 17. 第二阶段方向与后续路线图

MVP 稳定运行一至两周后，才评估：组织与数字化专题的自动选题、宏观音频推荐、脑科学自检的私人反馈层、人工反馈闭环、HTML 阅读页、周报、私有上下文仓库及 PDF/Word 导出。

还要记录一个明确的第二阶段方向：ChatGPT 最终审稿可能发现上游规则错误，但普通 ChatGPT GitHub 连接不能作为可靠写入通道。届时可让 Codex 参与结构化审核反馈的处理与回归修复；自动写回若要实现，必须使用受控的单用途反馈接口，且只能写入类似 `feedback/chatgpt-review/` 的隔离目录，不能让 ChatGPT 直接修改事件库、工作流、代码或 Secrets。MVP 不实现这个闭环。
